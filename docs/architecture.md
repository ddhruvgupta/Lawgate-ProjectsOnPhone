# Architecture

## Overview

Lawgate uses Clean Architecture (Onion Architecture) with four layers:

```
LegalDocSystem.API           → Controllers, middleware, Program.cs
LegalDocSystem.Application   → Services, DTOs, interfaces (business logic)
LegalDocSystem.Domain        → Entities, enums (pure business model)
LegalDocSystem.Infrastructure → EF Core, migrations, Azure Blob, background jobs
```

Dependency rule: outer layers depend inward. Domain has no dependencies. Infrastructure implements interfaces defined in Application.

---

## Backend Layer Details

### Domain (`LegalDocSystem.Domain`)
Pure C# classes with no framework dependencies.

**Entities:**
- `Company` — tenant root; all other entities reference `CompanyId`
- `User` — company employee with a `UserRole` enum; stores `PasswordHash` and `RefreshToken`
- `Project` — legal case or contract, scoped to a company
- `Document` — file metadata; the actual file lives in Azure Blob Storage
- `ProjectPermission` — grants a user a role on a specific project
- `AuditLog` — immutable record of user actions

**Enums:**
- `UserRole`: `CompanyOwner`, `Admin`, `User`, `Viewer`
- `ProjectStatus`: `Planning`, `Active`, `OnHold`, `Completed`, `Cancelled`, `Archived`
- `DocumentType`: `Contract`, `Brief`, `Motion`, `Pleading`, and others
- `SubscriptionTier`: `Trial`, `Basic`, `Professional`, `Enterprise`
- `PermissionLevel`: `None`, `Viewer`, `Commenter`, `Editor`, `Admin`

### Application (`LegalDocSystem.Application`)
Interfaces and service implementations consumed by the API layer.

**Interfaces:**
- `IAuthService` / `IJwtTokenService`
- `ICompanyService`, `IUserService`, `IProjectService`, `IDocumentService`
- `IBlobStorageService`

**Services** (implemented in Infrastructure or Application as appropriate):
- `AuthService` — register, login, refresh token
- `JwtTokenService` — issues and validates JWTs
- `CompanyService`, `UserService`, `ProjectService`, `DocumentService`
- `AzureBlobStorageService` — upload, chunked upload, SAS URL generation

### Infrastructure (`LegalDocSystem.Infrastructure`)
- `ApplicationDbContext` — EF Core context with all `DbSet<T>` properties
- `DbSeeder` — seeds demo data on startup in development
- `Migrations/` — EF Core migration history
- `AzureBlobStorageService` — Azure SDK integration
- `DocumentCleanupService` — `IHostedService` background job

### API (`LegalDocSystem.API`)
- Controllers: `AuthController`, `CompanyController`, `UserController`, `ProjectController`, `DocumentController`, `TestController`
- `GlobalExceptionMiddleware` — catches unhandled exceptions; returns structured JSON
- `Program.cs` — DI registration, JWT config, CORS, Serilog, auto-migration on startup

## Multi-Tenancy

Every entity that holds company data has a `CompanyId` foreign key. The JWT issued on login contains a `CompanyId` claim. Controllers extract this claim and pass it to services, which scope all queries to that company.

There is no global EF Core query filter for `CompanyId` — tenant isolation is enforced manually in each service method.

## Database Schema

All tables use `int` primary keys generated by the database (EF Core convention).

```
Companies (tenant root)
├── Users          (FK: CompanyId)
├── Projects       (FK: CompanyId)
│   ├── Documents          (FK: ProjectId, CompanyId)
│   └── ProjectPermissions (FK: ProjectId, UserId)
└── AuditLogs      (FK: CompanyId, UserId)
```

### Key columns

**Companies**
- `Id`, `Name`, `Email`, `Phone`, `Address`, `City`, `State`, `Country`, `PostalCode`
- `SubscriptionTier`, `SubscriptionStartDate`, `SubscriptionEndDate`
- `StorageUsedBytes`, `StorageQuotaBytes`
- `IsActive`, `CreatedAt`, `UpdatedAt`, `CreatedBy`

**Users**
- `Id`, `CompanyId`, `FirstName`, `LastName`, `Email`, `PasswordHash`, `Phone`
- `Role` (UserRole enum stored as int)
- `RefreshToken`, `RefreshTokenExpiry`
- `IsActive`, `CreatedAt`, `UpdatedAt`, `CreatedBy`

**Projects**
- `Id`, `CompanyId`, `Name`, `Description`, `ClientName`, `CaseNumber`
- `Status` (ProjectStatus enum), `StartDate`, `EndDate`
- `IsActive`, `CreatedAt`, `UpdatedAt`, `CreatedBy`

**Documents**
- `Id`, `ProjectId`, `CompanyId`, `FileName`, `BlobPath`, `ContentType`, `FileSizeBytes`
- `Status` (DocumentStatus enum — added in `AddDocumentStatus` migration)
- `UploadedByUserId`, `ParentDocumentId` (for versioning)
- `IsDeleted`, `DeletedAt`, `DeletedBy`, `CreatedAt`, `UpdatedAt`

## Authentication Flow

1. `POST /api/auth/register` — creates Company + User, returns JWT + refresh token
2. `POST /api/auth/login` — verifies BCrypt hash, returns JWT + refresh token
3. Client stores tokens; sends `Authorization: Bearer <token>` on subsequent requests
4. `POST /api/auth/refresh` — issues a new JWT using a valid refresh token
5. Refresh token is stored hashed in `Users.RefreshToken` with an expiry timestamp
6. JWT payload: `sub` (UserId), `email`, `companyId`, `role`, `exp`

## Frontend Architecture

```
frontend/src/
├── pages/        LoginPage.tsx, RegisterPage.tsx, DashboardPage.tsx
├── components/   Layout.tsx, ProtectedRoute.tsx
├── contexts/     AuthContext.tsx   (JWT storage + user state)
├── hooks/        useApi.ts         (Axios wrapper with auth header)
├── services/     api.ts            (API endpoint functions)
├── utils/        cn.ts             (clsx wrapper), formatters.ts
└── types/        TypeScript interfaces
```

React Router v7 handles routing. `ProtectedRoute` redirects unauthenticated users to `/login`. `AuthContext` persists the token to `localStorage` and exposes `user`, `login`, `logout`.

## Azure Blob Storage

Blob path structure: `{companyId}/{projectId}/{documentId}/{filename}`

Documents are never served directly. The API generates short-lived SAS URLs (via `AzureBlobStorageService`) that the client uses to upload or download.

Local development uses Azurite (Azure Storage emulator) running in Docker on ports 10000–10002.
